# v2.1 ‚Äî Faithful Backlog (quality) ‚Äî Design

> Status: **Draft** ¬∑ Date: 2026-07-02 ¬∑ Follows the first full live run on Bloom (274 issues ‚Üí Jira BLM).
> Scope: make the generated backlog a *faithful, actionable* picture of the project. No new pipeline; sharpen what already exists.

---

## 1. What the live run revealed

The first end-to-end `/eng:run` on Bloom produced a **grounded but not faithful** backlog:

- Epics/stories mapped to **real Bloom features** with –¢–ó numbers (BLM-24 Technical Debt, BLM-40 Public online booking –¢–ó-12, BLM-7 Multi-shop –¢–ó-7.4). Grounding is good.
- But: **nothing that is already done** appears; **priority is all Medium**; **descriptions are mixed RU/UK/EN**; tasks carry **no discipline tag** (unclear who to assign); the **timeline is empty**; **~70% are sub-tasks** (possibly over-decomposed).

## 2. The core insight (drives most of v2.1)

Most of these are **not missing analysis ‚Äî they are under-projection.** The model already computes `category`, `status`, `phase`, `priority`; they just never reach Jira. So a large part of v2.1 is *rendering existing model data into the right Jira fields*, which is cheap and high-ROI. Only a few items need new logic.

## 3. Guiding principles

1. **Accuracy before performance.** (v2.1 = faithful output; freshness/cost = later.)
2. **Jira is where humans work.** Re-sync must not clobber human edits ‚Üí conservative updates.
3. **Deterministic over LLM-guessed metadata.** Priority/sequence derive from `phase`/git, not model opinion.
4. **English output** ‚Äî the platform is neutral; English is the common language.
5. **Two axes, two fields.** *Discipline* ("who does it") ‚âÝ *decision-state* ("is it decided"). Never conflate into one field.

## 4. Items

Each: **problem ‚Üí root cause (where) ‚Üí fix ‚Üí notes**.

### A. Show what is already done (epic status map) ‚Äî highest impact
- **Problem:** for a mature repo (Bloom, mostly shipped), the tracker looks like nothing is built. The done half of the analysis is discarded.
- **Root cause:** `build-project-model` SKILL expands only `Planned` / `Partially Implemented`; `Implemented` domains are recorded but never surfaced.
- **Fix:** create **epics for all major domains, including done ones**, carrying an epic-level `status` (`done` / `in-progress` / `todo`) from `knowledgeModel.domains[].status`. **Only expand stories/tasks under not-done epics.** So Jira shows the full product map with status, without hundreds of closed sub-tasks.
  - `planning-model.mjs`: add `status` to epics; keep expanding stories only for non-done.
  - `sync-tracker`: after creating a done epic, **transition it to Done** (next-gen statuses come via transitions ‚Äî create then `transitionJiraIssue`).
- **Notes:** epic model currently has no `status` field ‚Äî add it. Transition-to-Done is create-then-transition (extra call), not a create field. **Depends on K (reality scan)** ‚Äî a mock/stub marked `Done` is worse than unmapped; done-detection reliability gates this item.

### B. English descriptions
- **Problem:** mixed RU/UK/EN output.
- **Root cause:** agent briefs and the build SKILL don't specify output language ‚Üí agents mirror the repo (Bloom docs are UK/RU).
- **Fix:** force **English** for all generated titles/descriptions/AC in `agents/*.md` and the `analyze`/`build` SKILL prompts. (Analysis still *reads* any language; it *writes* English.)
- **Notes:** cheapest high-visibility win.

### C. Discipline tags on tasks
- **Problem:** "–Ω–µ–ø–æ–Ω—è—Ç–Ω–æ —á—Ç–æ –∏ –Ω–∞ –∫–æ–≥–æ –∞—Å—Å–∞–π–Ω–∏—Ç—å" ‚Äî tasks have no `[BE]`/`[FE]`/‚Ä¶ marker.
- **Root cause:** `task.category` exists in the model but is never rendered to Jira.
- **Fix:** prefix the Jira summary with the discipline from `category` and add a matching label: `[BE] Update EF‚Ä¶`, `[FE] Rework calendar`, `[DB] Add index‚Ä¶`, `[QA] ‚Ä¶`, `[Design] ‚Ä¶`, `[Admin] ‚Ä¶`.
  - `sync-tracker` create/update: build summary as `[TAG] <title>` + label `disc:<category>`.
- **Notes:** extend the taxonomy ‚Äî see D.

### D. Two axes: discipline (`category`) vs decision-state (`needs-decision`)
- **Problem:** user wants categories like `admin`, `design`, and "if we haven't decided ‚Üí project ownership".
- **Critical distinction:** `admin`/`design` are **disciplines**; "not decided ‚Üí PO" is a **state**, not a discipline. A task can be `[BE]` *and* undecided ‚Äî one field can't hold both.
- **Fix:**
  - Extend `category` (discipline) taxonomy: add `admin`, `design`/`ux` (alongside backend/frontend/database/validation/tests/documentation/logging/monitoring/migration). Update `store.mjs` `TASK_CATEGORIES` + `schemas/project-model.schema.json` enum.
  - Add a **separate** boolean/flag `needsDecision` on story/task ‚Üí Jira label `needs-decision` + a visible `[?]` marker. This is the "project ownership / unresolved" bucket.
- **Notes:** keep the two orthogonal; do not merge into `category`.

### E. Priority derived from phase (deterministic)
- **Problem:** all 202+ issues are Medium.
- **Root cause:** (1) priority lives only per-domain in `planningModel.items`, not on backlog nodes; (2) `sync-tracker` never sets the Jira `priority` field ‚Üí Jira default (Medium).
- **Fix:** derive priority **deterministically from `phase`** (not LLM-guessed): `MVP ‚Üí High`, `Production Ready ‚Üí Medium`, `Public Release`/`Scaling ‚Üí Low`, `Enterprise`/`AI ‚Üí Lowest`. Stamp it on epics/stories in `planning-model.mjs`; map it in `sync-tracker` create/update.
- **Notes:** "leave everything Medium, human triages" is a defensible alternative ‚Äî but derived-from-phase is a better, honest default. Never let an agent free-assign priority (‚Üí everything-High noise).

### F. Timeline
- **Problem:** timeline is empty (no dates).
- **Split (user-corrected):**
  - **Done work ‚Üí real dates from git.** It is history, not fiction: repo start, roadmap-update commits, feature-completion commits.
  - **Future work ‚Üí sequencing only**, ordered by `phase` + `dependsOn`. **No fabricated calendar deadlines.**
- **Fix:** for done epics, derive a start/end from git ‚Äî `git log --follow` over the domain's `sources[]` ‚Üí first/last commit dates ‚Üí Jira start/due. Future epics: order by phase/deps (no due dates, or phase-block bands).
- **Notes:** this is **more than rendering** ‚Äî needs a `feature ‚Üí commit-range` derivation (KM stores `sources` files, not commit ranges). Likely a small `git-dates` helper. Confirm next-gen date fields are writable via the Rovo MCP.

### G. Conservative idempotent re-sync (backfill without clobbering)
- **Problem:** re-run must **update** the existing 274 issues (English, prefixes, priority, status) ‚Äî not no-op ‚Äî yet must not destroy human edits.
- **Root cause:** current `sync-tracker` update path is vague ("push same fields").
- **Fix:** on `update`:
  - **Always** (safe, additive): set/refresh labels, summary discipline-prefix, priority, epic status.
  - **Descriptions:** overwrite only if unchanged since last eng-sync **or** empty; otherwise **warn and skip** (offer `--force-descriptions`). Track a per-issue "last synced by eng" marker (e.g., a label/property with the model hash or commit).
- **Notes:** **The first Bloom re-sync is safe** ‚Äî nobody has hand-edited BLM yet ‚Äî so it can run without the guard. But the guard is required before this ships for repeated use. Reconciliation itself already works (eng-id labels ‚Üí 0 dups, proven in the crash-recovery during the first run).

### H. Sub-task granularity
- **Problem:** ~70% of issues are sub-tasks ‚Äî possibly over-decomposed / filler.
- **Fix:** tighten `build` guidance ‚Äî a sub-task must be **independently meaningful and assignable**; no "—Ä–∞–¥–∏ –≥–∞–ª–æ—á–∫–∏" splits. Consider a cap / a reviewer check on subtask-to-story ratio.
- **Notes:** softer, judgment-based; validate against the real BLM data before tuning hard.

### I. UI/UX review agent (+ security agent)
- **Problem:** no dedicated UI/UX lens; security is only a devops side-bullet, yet Bloom holds client PII.
- **Fix:** add two agents to the panel, with **clean lanes** and **grounded findings** (cite files; absence = `risk: unknown` "no evidence", never asserted):
  - `ux-reviewer` ‚Äî static frontend/UX/consistency/a11y review of Razor components ‚Üí `[FE]`/`[Design]` tasks.
  - `security-engineer` ‚Äî auth/authz, secrets, PII, encryption, tenant-isolation/RLS, injection. Remove the generic "security posture" bullet from `devops-engineer` to avoid overlap.
- **Notes:** every extra agent = another full pass = **more cost per run**. So couple this with a **configurable panel** (core always; security/ux/data on deep passes). Real *visual* UX audit needs the **running app + screenshots** ‚Äî out of scope here (static ceiling only).
- **Business-analyst agent ‚Äî considered and REJECTED (evidence-based, 02.07).** A JQL check of the live BLM backlog shows business rules are already captured precisely by the current panel: BLM-56 commission rules per master/service, BLM-140 validation (percent 0‚Äì100, fixed ‚â• 0, no overlapping rules), BLM-178/277 booking-overlap exclusion constraint, BLM-210 overlap-rejection test, BLM-183/245/247 no-show/refund/forfeiture policy matrix, BLM-240 discount-in-price-calc. A BA lens would duplicate PO (scope) / qa-lead (AC) / data-engineer lanes. The user also maintains a separate BA skill. **Do not add.** The selection rule stands: add a lens only if it yields findings no existing agent does.

### K. Completeness / reality scan ‚Äî done-detection depth (foundational)
- **Problem (found 02.07 by user):** the analysis missed things that ARE visible in code: a **dashboard on mock data** (marked as if done) and a **table with no sort/filter/search** (real but feature-poor). Neither reached the tickets. The done/not-done map that A relies on is therefore not trustworthy.
- **Two distinct miss types ‚Üí two mechanisms:**
  - **Fake-done (mock/stub/TODO)** ‚Äî detectable by **hard signals**. Add a deterministic **completeness scan** (a script): grep the repo for `Mock`, `TODO`, `FIXME`, `NotImplemented`, `NotSupportedException`, `throw new NotImplemented`, `Noop*`, hardcoded/seed arrays rendered in UI, `// placeholder`. Feed the hits to the agents as "verify these; downgrade status if fake." Sharpen `senior-engineer`'s brief: **"renders ‚âÝ done"** ‚Äî actively hunt stub-vs-real, don't assume a component that compiles is finished. (It caught Noop senders ‚Üí BLM-4, but missed the mock dashboard ‚Äî the mandate exists; the depth failed.)
  - **Feature-poor (real but incomplete)** ‚Äî *absence* of expected features (table without sort/filter/search) is **not greppable**; needs the **`ux-reviewer`** semantic read of the component ("renders a table, no filter/sort/search handlers ‚Üí gap"). This is the concrete evidence for item I.
- **Honest limit:** absence-detection is inherently hard ‚Äî the model must *know* a table "should" have a filter (a UX expectation), so coverage will be imperfect even with a UX lens.
- **Dependency:** **A must not ship before K.** Mapping a mock as `Done` in Jira is worse than leaving it unmapped ‚Äî the tracker becomes confidently wrong. Done-detection reliability is the foundation of the whole "faithful backlog" goal.

### L. Semantic domain dedup (final-reviewer gap)
- **Problem (found 02.07):** analyze produced ~53 domains with **semantic
  duplicates** ‚Äî string-similar (`cash-desk`/`cashdesk`/`finance-cashdesk`,
  `fiscalization`/`cash-desk-fiscalization`) AND meaning-similar
  (`Bookings` / `Bookings & Calendar` / `Calendar UI`; `Roles/permissions` /
  `Capability-based authz`). The done-map surfaced ~15 duplicate Done epics out
  of 24. The id-merge dedups by exact id; `final-reviewer` was mandated to catch
  "duplicates the id-merge missed" but did not.
- **Fix:** strengthen `final-reviewer` (runs inside `analyze-project`) to merge
  semantically-duplicate domains into a canonical one. A deterministic pre-pass
  can catch string-similar ids; the meaning-similar ones need the LLM lens.
- **Scope note:** this lives in **analyze**, so it only helps a **fresh
  re-analysis** ‚Äî NOT `refresh-model` (deterministic, no agents). For an existing
  model the dedup is a one-time interactive collapse before refresh.

### P. Sync transitions by status CATEGORY, not name (template robustness) ‚Äî 1.0 scope
- **Problem (found 02.07, BLM architecture review):** `sync-tracker` resolves the
  Done/In-Progress transition by status **name** (`Done` / `In Progress`). Fragile:
  breaks on renamed/localized statuses (we already hit locale issues with issue-type
  names). For a project used as the **default template on every repo**, names vary
  but Jira's **three status categories are universal** (To Do / In Progress / Done).
- **Fix:** in `sync-tracker`, pick the transition whose target `statusCategory.key`
  matches the intent (`done` / `indeterminate` / `new`), not the status name. Small
  testable helper (given the `getTransitions` list + a category key ‚Üí the transition
  id). Robust across any template.

### Q. setup-toolkit: workflow/status-category health check + recommended template ‚Äî 1.0 scope
- **Problem (BLM review):** BLM's statuses **Backlog** and **To Do** are miscategorized
  into the **In Progress** category ‚Üí not-started epics report as "in progress";
  category-based reports/board colors lie. Also redundant not-started statuses
  (Backlog + Idea + To Do) and an unused `Feature` issue type.
- **Fix (verify-only ‚Äî never auto-reconfigure the user's Jira):**
  - `setup-toolkit` reads statuses+categories and **WARNS** on: a status whose
    category looks wrong (e.g. a "Backlog"/"To Do" named status in the In-Progress
    category), >1 not-started status, or a target category with no status.
  - `references/providers/jira.md` documents the **recommended template**: issue
    types **Epic / Story / Task / Sub-task / Bug** (drop `Feature`, English names),
    and a lean workflow **To Do ‚Üí In Progress ‚Üí Done** (or Backlog ‚Üí To Do ‚Üí In
    Progress ‚Üí In Review ‚Üí Done if grooming+QA), 1 status = 1 column, each in its
    correct category.
- **Out of scope (by design):** editing statuses/workflow/categories ‚Äî neither the
  toolkit nor the Rovo MCP can (no admin tools); that is a manual Jira-UI fix.

### O. Translate-in-place (English on existing issues) ‚Äî 1.0 scope
- **Problem (confirmed 02.07, BLM-12):** existing issues keep mixed-language text
  from the first run ‚Äî especially **Acceptance Criteria** (folded into the
  description, which the conservative-update rule deliberately does not overwrite).
  B (English) only affects *new* generation.
- **Correction:** this is **decoupled** from the "stable ids" v2.2 item. That item
  is about a full **re-draft** (which regenerates ids ‚Üí duplicates).
  **Translate-in-place does NOT re-draft** ‚Äî it rewrites only the text fields
  (title / description / AC given-when-then) to English while **keeping every id,
  trackerKey and the structure**. So it is safe (no duplicates) and belongs in
  **1.0**, not v2.2.
- **Fix:** a translation pass over the stored model's text (an LLM step, cheaper
  than re-analysis ‚Äî no code reading), then a `sync-tracker` update that pushes the
  translated text with a **translation override** (allowed to overwrite a
  description that is still toolkit-owned / unchanged since last eng-sync; still
  skips human-edited ones). Result: existing BLM issues become fully English
  without wipe or duplicates.

### M. Analyze completeness for documented planned work
- **Problem (found 02.07, roadmap comparison):** the done-map matched the human
  roadmap ~1:1, but analyze **missed a documented planned feature** ‚Äî "multi-service
  booking" (multiple services per booking) ‚Äî even though it is in the repo
  (`docs/next-session.md`, `docs/architecture/*`, an admin-feedback spec). So the
  gap is **coverage depth in analyze**, NOT an outside-repo input problem. The
  product-owner agent under-extracted forward plans that live in docs but not yet
  in code.
- **Fix:** a **completeness critic** pass ‚Äî after the panel, ask "which documented
  plans / roadmap items / TODO-features have no domain or backlog entry?" and
  surface the misses (the completeness-critic pattern). Sharpen `product-owner`:
  extract planned-but-not-coded features from roadmap/next-session/spec docs, not
  only code-derived domains.

### N. Jira as team source of truth (v3 direction, not v2.1)
- **Direction (user, 02.07):** deprecate the local/OneDrive roadmap; the team
  (now includes QA) runs everything from Jira. Toolkit's role shifts from
  "generate the backlog once" to "keep Jira honest about code reality (done-map,
  tech-debt, drift) while humans own forward planning IN Jira."
- **Already supported:** the toolkit only manages issues carrying its `eng-id` /
  `eng-sync` labels, so **team-added Jira issues (no eng-id) are never touched** on
  re-sync ‚Äî coexistence works by design.
- **The design question to nail:** **field ownership.** Which fields does the
  toolkit re-assert on its own issues (done-status from code, priority-from-phase,
  discipline tags) vs which are human-owned and never overwritten (assignee,
  sprint, workflow status beyond Done, comments)? Define the contract so re-sync
  and human team-management coexist without stomping each other. This is the core
  of "Jira as living source of truth + toolkit reconcile."

## 5. Priority within v2.1

1. **Cheap rendering wins (do first, independent):** B (English), C (discipline tags), E (priority-from-phase), D (decision-state flag + taxonomy). These are safe on the current backlog and validated by one re-sync.
2. **Foundational accuracy (gates A):** K (reality/completeness scan) + the `ux-reviewer` half of I. Done-detection must be trustworthy before mapping "done" to Jira.
3. **Then:** A (done epic-status map), G (conservative update-backfill).
4. **Bigger / later in v2.1:** F (git dates for done), H (subtask granularity), the `security-engineer` half of I + configurable panel.

## 6. Explicitly deferred (v2.2+ / v3) ‚Äî not this version

- **v2.2 ‚Äî Confluence + documentation sync** (`sync-docs` skill ‚Üí `createConfluencePage`): project overview, the Knowledge Model as a living page, and **Engineering Memory** (ADR-as-first-class-input + a `decision-drift` risk type ‚Äî the "why", e.g. Outbox-not-Kafka). Confluence is the natural home for "what's done" + "why" that don't fit Jira issues. **Not a priority now.**
- **v2.2 ‚Äî re-run update-in-place (stable ids).** Found 02.07: a full re-run that
  changes text (e.g. English re-titling) **duplicates** the agent-drafted issues ‚Äî
  `eng-id` reconcile matches on node ids, but re-drafting regenerates ids (and the
  story set), so nothing matches ‚Üí creates new + orphans old. Deterministic epics
  (done/tech-debt/bugs) are safe; agent-drafted ones are not. To make production
  re-runs non-destructive: (a) stable, content-independent node ids, AND (b) a
  **translate-in-place** pass that rewrites existing model text (to English) while
  keeping ids/trackerKeys, so `sync` *updates* the same issues instead of creating
  new ones. Until then, changing existing text = wipe + regenerate (fine for a
  test project, not for production).
- **v2.2 ‚Äî coarse freshness-skip in `run`** (`commit unchanged ‚Üí skip analyze`) and a **configurable agent panel** for cost. Note: **incremental KM merge is rejected** ‚Äî the union-merge can't update a changed `status` (implemented‚Üípartial silently lost); a true reconciliation engine is Git-merge-hard and not worth it. Coarse skip only.
- **v2.3 ‚Äî derived read-only projections** (Dependency Graph, Feature Registry) ‚Äî **only when a consumer needs them**; must be `derived-never-authored`, regenerated not persisted-and-edited (else they become a second source of truth).
- **v3 ‚Äî execution** (frontend-design generation, PRs, tests). Different product ("Analysis ‚Üí Knowledge ‚Üí Execution"); decide *deep vs wide* before entering.
