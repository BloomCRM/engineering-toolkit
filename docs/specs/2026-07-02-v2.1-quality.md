# v2.1 — Faithful Backlog (quality) — Design

> Status: **Draft** · Date: 2026-07-02 · Follows the first full live run on Bloom (274 issues → Jira BLM).
> Scope: make the generated backlog a *faithful, actionable* picture of the project. No new pipeline; sharpen what already exists.

---

## 1. What the live run revealed

The first end-to-end `/eng:run` on Bloom produced a **grounded but not faithful** backlog:

- Epics/stories mapped to **real Bloom features** with ТЗ numbers (BLM-24 Technical Debt, BLM-40 Public online booking ТЗ-12, BLM-7 Multi-shop ТЗ-7.4). Grounding is good.
- But: **nothing that is already done** appears; **priority is all Medium**; **descriptions are mixed RU/UK/EN**; tasks carry **no discipline tag** (unclear who to assign); the **timeline is empty**; **~70% are sub-tasks** (possibly over-decomposed).

## 2. The core insight (drives most of v2.1)

Most of these are **not missing analysis — they are under-projection.** The model already computes `category`, `status`, `phase`, `priority`; they just never reach Jira. So a large part of v2.1 is *rendering existing model data into the right Jira fields*, which is cheap and high-ROI. Only a few items need new logic.

## 3. Guiding principles

1. **Accuracy before performance.** (v2.1 = faithful output; freshness/cost = later.)
2. **Jira is where humans work.** Re-sync must not clobber human edits → conservative updates.
3. **Deterministic over LLM-guessed metadata.** Priority/sequence derive from `phase`/git, not model opinion.
4. **English output** — the platform is neutral; English is the common language.
5. **Two axes, two fields.** *Discipline* ("who does it") ≠ *decision-state* ("is it decided"). Never conflate into one field.

## 4. Items

Each: **problem → root cause (where) → fix → notes**.

### A. Show what is already done (epic status map) — highest impact
- **Problem:** for a mature repo (Bloom, mostly shipped), the tracker looks like nothing is built. The done half of the analysis is discarded.
- **Root cause:** `build-project-model` SKILL expands only `Planned` / `Partially Implemented`; `Implemented` domains are recorded but never surfaced.
- **Fix:** create **epics for all major domains, including done ones**, carrying an epic-level `status` (`done` / `in-progress` / `todo`) from `knowledgeModel.domains[].status`. **Only expand stories/tasks under not-done epics.** So Jira shows the full product map with status, without hundreds of closed sub-tasks.
  - `planning-model.mjs`: add `status` to epics; keep expanding stories only for non-done.
  - `sync-tracker`: after creating a done epic, **transition it to Done** (next-gen statuses come via transitions — create then `transitionJiraIssue`).
- **Notes:** epic model currently has no `status` field — add it. Transition-to-Done is create-then-transition (extra call), not a create field.

### B. English descriptions
- **Problem:** mixed RU/UK/EN output.
- **Root cause:** agent briefs and the build SKILL don't specify output language → agents mirror the repo (Bloom docs are UK/RU).
- **Fix:** force **English** for all generated titles/descriptions/AC in `agents/*.md` and the `analyze`/`build` SKILL prompts. (Analysis still *reads* any language; it *writes* English.)
- **Notes:** cheapest high-visibility win.

### C. Discipline tags on tasks
- **Problem:** "непонятно что и на кого ассайнить" — tasks have no `[BE]`/`[FE]`/… marker.
- **Root cause:** `task.category` exists in the model but is never rendered to Jira.
- **Fix:** prefix the Jira summary with the discipline from `category` and add a matching label: `[BE] Update EF…`, `[FE] Rework calendar`, `[DB] Add index…`, `[QA] …`, `[Design] …`, `[Admin] …`.
  - `sync-tracker` create/update: build summary as `[TAG] <title>` + label `disc:<category>`.
- **Notes:** extend the taxonomy — see D.

### D. Two axes: discipline (`category`) vs decision-state (`needs-decision`)
- **Problem:** user wants categories like `admin`, `design`, and "if we haven't decided → project ownership".
- **Critical distinction:** `admin`/`design` are **disciplines**; "not decided → PO" is a **state**, not a discipline. A task can be `[BE]` *and* undecided — one field can't hold both.
- **Fix:**
  - Extend `category` (discipline) taxonomy: add `admin`, `design`/`ux` (alongside backend/frontend/database/validation/tests/documentation/logging/monitoring/migration). Update `store.mjs` `TASK_CATEGORIES` + `schemas/project-model.schema.json` enum.
  - Add a **separate** boolean/flag `needsDecision` on story/task → Jira label `needs-decision` + a visible `[?]` marker. This is the "project ownership / unresolved" bucket.
- **Notes:** keep the two orthogonal; do not merge into `category`.

### E. Priority derived from phase (deterministic)
- **Problem:** all 202+ issues are Medium.
- **Root cause:** (1) priority lives only per-domain in `planningModel.items`, not on backlog nodes; (2) `sync-tracker` never sets the Jira `priority` field → Jira default (Medium).
- **Fix:** derive priority **deterministically from `phase`** (not LLM-guessed): `MVP → High`, `Production Ready → Medium`, `Public Release`/`Scaling → Low`, `Enterprise`/`AI → Lowest`. Stamp it on epics/stories in `planning-model.mjs`; map it in `sync-tracker` create/update.
- **Notes:** "leave everything Medium, human triages" is a defensible alternative — but derived-from-phase is a better, honest default. Never let an agent free-assign priority (→ everything-High noise).

### F. Timeline
- **Problem:** timeline is empty (no dates).
- **Split (user-corrected):**
  - **Done work → real dates from git.** It is history, not fiction: repo start, roadmap-update commits, feature-completion commits.
  - **Future work → sequencing only**, ordered by `phase` + `dependsOn`. **No fabricated calendar deadlines.**
- **Fix:** for done epics, derive a start/end from git — `git log --follow` over the domain's `sources[]` → first/last commit dates → Jira start/due. Future epics: order by phase/deps (no due dates, or phase-block bands).
- **Notes:** this is **more than rendering** — needs a `feature → commit-range` derivation (KM stores `sources` files, not commit ranges). Likely a small `git-dates` helper. Confirm next-gen date fields are writable via the Rovo MCP.

### G. Conservative idempotent re-sync (backfill without clobbering)
- **Problem:** re-run must **update** the existing 274 issues (English, prefixes, priority, status) — not no-op — yet must not destroy human edits.
- **Root cause:** current `sync-tracker` update path is vague ("push same fields").
- **Fix:** on `update`:
  - **Always** (safe, additive): set/refresh labels, summary discipline-prefix, priority, epic status.
  - **Descriptions:** overwrite only if unchanged since last eng-sync **or** empty; otherwise **warn and skip** (offer `--force-descriptions`). Track a per-issue "last synced by eng" marker (e.g., a label/property with the model hash or commit).
- **Notes:** **The first Bloom re-sync is safe** — nobody has hand-edited BLM yet — so it can run without the guard. But the guard is required before this ships for repeated use. Reconciliation itself already works (eng-id labels → 0 dups, proven in the crash-recovery during the first run).

### H. Sub-task granularity
- **Problem:** ~70% of issues are sub-tasks — possibly over-decomposed / filler.
- **Fix:** tighten `build` guidance — a sub-task must be **independently meaningful and assignable**; no "ради галочки" splits. Consider a cap / a reviewer check on subtask-to-story ratio.
- **Notes:** softer, judgment-based; validate against the real BLM data before tuning hard.

### I. UI/UX review agent (+ security agent)
- **Problem:** no dedicated UI/UX lens; security is only a devops side-bullet, yet Bloom holds client PII.
- **Fix:** add two agents to the panel, with **clean lanes** and **grounded findings** (cite files; absence = `risk: unknown` "no evidence", never asserted):
  - `ux-reviewer` — static frontend/UX/consistency/a11y review of Razor components → `[FE]`/`[Design]` tasks.
  - `security-engineer` — auth/authz, secrets, PII, encryption, tenant-isolation/RLS, injection. Remove the generic "security posture" bullet from `devops-engineer` to avoid overlap.
- **Notes:** every extra agent = another full pass = **more cost per run**. So couple this with a **configurable panel** (core always; security/ux/data on deep passes). Real *visual* UX audit needs the **running app + screenshots** — out of scope here (static ceiling only).
- **Business-analyst agent — considered and REJECTED (evidence-based, 02.07).** A JQL check of the live BLM backlog shows business rules are already captured precisely by the current panel: BLM-56 commission rules per master/service, BLM-140 validation (percent 0–100, fixed ≥ 0, no overlapping rules), BLM-178/277 booking-overlap exclusion constraint, BLM-210 overlap-rejection test, BLM-183/245/247 no-show/refund/forfeiture policy matrix, BLM-240 discount-in-price-calc. A BA lens would duplicate PO (scope) / qa-lead (AC) / data-engineer lanes. The user also maintains a separate BA skill. **Do not add.** The selection rule stands: add a lens only if it yields findings no existing agent does.

## 5. Priority within v2.1

1. **Cheap rendering wins (do first):** B (English), C (discipline tags), E (priority-from-phase), D (decision-state flag + taxonomy).
2. **Medium:** A (done epic-status map), G (conservative update-backfill).
3. **Needs derivation / bigger:** F (git dates for done), I (ux + security agents + configurable panel), H (subtask granularity).

## 6. Explicitly deferred (v2.2+ / v3) — not this version

- **v2.2 — Confluence + documentation sync** (`sync-docs` skill → `createConfluencePage`): project overview, the Knowledge Model as a living page, and **Engineering Memory** (ADR-as-first-class-input + a `decision-drift` risk type — the "why", e.g. Outbox-not-Kafka). Confluence is the natural home for "what's done" + "why" that don't fit Jira issues. **Not a priority now.**
- **v2.2 — coarse freshness-skip in `run`** (`commit unchanged → skip analyze`) and a **configurable agent panel** for cost. Note: **incremental KM merge is rejected** — the union-merge can't update a changed `status` (implemented→partial silently lost); a true reconciliation engine is Git-merge-hard and not worth it. Coarse skip only.
- **v2.3 — derived read-only projections** (Dependency Graph, Feature Registry) — **only when a consumer needs them**; must be `derived-never-authored`, regenerated not persisted-and-edited (else they become a second source of truth).
- **v3 — execution** (frontend-design generation, PRs, tests). Different product ("Analysis → Knowledge → Execution"); decide *deep vs wide* before entering.
